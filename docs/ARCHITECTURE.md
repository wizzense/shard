# Shard architecture

## The shape of the problem

Pipeline parallelism. A model is a stack of transformer layers. Split the stack into N contiguous blocks, put one block on each GPU. A token is generated by:

```
prompt -> embed -> node0 (layers 0..k) -> node1 (layers k..2k) -> ... -> nodeN-1 (final layers) -> sample -> next token
```

Each node holds only its block in VRAM, runs its block's forward pass, and ships the resulting activations (a hidden-state tensor) to the next node. This is how a 200GB model fits on 4x 24GB cards.

Two things make this hard on a real swarm, and they are the whole reason Shard exists:

1. **Latency.** Autoregressive generation is one token at a time, and every token traverses the entire pipeline. With nodes on home internet at 50-80ms each, a 4-hop pipeline pays ~250-650ms per token round trip. That is 1-2 tok/s. Dead on arrival.

2. **Transport.** The activation tensor has to cross the public internet between two machines behind NAT, every step, in both directions (forward activations, backward for the KV-cache/sampling coordination). It has to be fast, reliable, and reconnect when a home connection blips. This is the layer that breaks in existing tools.

Shard attacks (1) with speculative decoding and (2) with a transport it owns.

## The five layers

### 1. Per-node runtime (`shard/node.py`)

Each node runs a serving engine for its block of layers. We do **not** rebuild attention kernels, paging, or quantization. We wrap **SGLang** (battle-tested, production-grade) and drive it at the block level:

- `load_shard(model, layer_range)` — pull only this block's weights (from HF or a c0mpute mirror) and load to VRAM
- `forward(hidden_states, kv_meta) -> hidden_states` — run this block, return activations for the next stage
- KV-cache lives per-node for its own layers; the coordinator tracks sequence ids

A node is just: a block of layers + a transport endpoint + a heartbeat.

### 2. Transport (`shard/transport.py`) — the part we own

This is the wedge. Requirements:

- **Direct P2P between adjacent stages** over QUIC (UDP, built-in congestion control, multiplexed streams, 0-RTT reconnect). Start with `aioquic` in Python to move fast, lift the hot path to Rust (`quinn`) when bandwidth demands.
- **NAT traversal**: a rendezvous service (c0mpute orchestrator at first) coordinates hole-punching between the two peers of each pipeline edge. Relay fallback for the symmetric-NAT cases that won't punch.
- **Activation codec**: the payload is a hidden-state tensor every step. Quantize it (fp8/int8) and optionally compress before sending. Activations tolerate quantization far better than weights, and bandwidth is the cost center on a home uplink.
- **Authenticated, encrypted, pickle-free wire** (shipped, phase 0: `phase0/wire.py`). Phase 0 sent each message as raw `pickle` over plaintext TCP — two problems at once: a passive observer on the path read the activations, and `recv_msg` ran `pickle.loads` on bytes from an untrusted socket (remote code execution). The wire now frames a message as a JSON header + raw tensor bytes — no pickle, so a hostile frame is a parse error, never code execution from any sender — and seals each frame with ChaCha20-Poly1305 under a shared `SHARD_PSK` (confidentiality + integrity + the sender holds the key; a forged or tampered frame fails the auth tag and the edge is reset). This seals the *line*; it does not change endpoint trust — a participating node still decrypts to run its layer (see Privacy). Key distribution is a pre-shared secret for now, right for a trusted/known deployment; a permissionless swarm needs per-node identities + a keyed handshake (Noise / QUIC-TLS keyed by `cwt_` tokens), the phase-1 follow-up.
- **Backpressure, timeouts, reconnect**: a stage that stalls must not wedge the pipeline. Edges are supervised, a dropped node is detected fast and reported to the scheduler for reassignment.

The design rule: this layer is observable and debuggable. Every edge logs its own health. No black-box binary that returns "broken pipe" with nothing behind it. That single property is what keeps a real swarm transport debuggable in production.

### 3. Speculative decoding coordinator (`shard/specdec.py`)

This is what makes WAN latency survivable, and it is the heart of the paper.

- The **entry node** runs a small **draft model** locally (1-3B, same tokenizer family as the target, uncensored). This is fast and pays no WAN cost.
- The draft proposes K candidate tokens cheaply.
- The **target model** (the big split one) verifies all K in a **single pipeline traversal**: feed the K draft tokens as one batch, every node runs its block once, and we get the target's distribution for all K positions in one round trip.
- Standard speculative-decoding acceptance: keep the longest prefix the target agrees with, resample the first rejected position, discard the rest.
- Net throughput is roughly (accepted tokens per round) / (one pipeline round-trip). Instead of 1 token per ~400ms, you get ~3-5 accepted tokens per ~400ms.
- **K is tuned adaptively** to the measured pipeline latency and the running acceptance rate. High latency link -> draft deeper (bigger K) to amortize more.

Reference numbers from the paper (arXiv 2602.16760): ~8-9 tok/s for a 7B-class target at 80ms links, 15-19 tok/s at 20ms, scaling with how good the draft is. Those are the numbers Shard is chasing, honestly measured.

### 4. Scheduler / control plane (`shard/scheduler.py`)

Light and replaceable. Responsibilities:

- **Layer allocation**: given the target model and the set of currently-joined GPUs (heterogeneous VRAM), fit the model: assign each node a contiguous layer block so every block fits in its node's VRAM, with the whole stack covered.
- **Topology**: order the nodes into the pipeline, prefer low-latency edges where the measured RTT graph allows.
- **Registry + health**: who is in the swarm, what block they hold, are they alive (heartbeat).
- **Reassignment**: a node drops mid-generation -> its block is reassigned to a spare or the pipeline is rebuilt, in-flight requests retried.

Hosted by the c0mpute orchestrator at first (pragmatic, ship faster). It holds no weights and no user data, so decentralizing it later (rotating/elected scheduler) is a clean follow-up, not a rewrite.

### 5. c0mpute integration

Shard is not a standalone toy, it is c0mpute's serving engine. Reuse what already exists:

- **Permissionless join**: one-line installer that auto-installs deps (SGLang, drivers check) the way the existing c0mpute workers do. Plug and play, the operator does nothing but run it.
- **Worker auth**: the existing `cwt_` worker tokens.
- **Payments**: per-token USDC metering on the existing rails. A node earns for the tokens its block helped produce.
- **Dashboard + referrals**: the existing c0mpute surfaces.

## Privacy

The open problem, and a direct test of pillar two.

The paper shows that a malicious node sitting in the pipeline can reconstruct **35-59%** of the user's tokens from the intermediate activations it sees. For a network that sells "private," that is not acceptable as-is. Options, roughly in order of how much I'd lean on them:

- **a. Boundary pinning (recommended core).** The embedding and the final few layers carry the most recoverable signal. Keep those blocks on operator-run or staked/trusted nodes; let untrusted volunteers hold only deep middle layers, which leak far less.
- **b. Trusted routing for sensitive requests.** Let a request opt into "trusted nodes only" (vetted/staked operators), trading some decentralization for privacy on demand.
- **c. Activation obfuscation.** Inject noise or permute the hidden dimension on each edge, with a shared transform the honest pipeline can invert. Degrades reconstruction; needs measuring against quality cost.
- **d. Honest disclosure + roadmap.** Ship phase by phase with the current guarantee documented, never overclaim. "Private" earns its word as the hardening lands, it is not asserted on day one.

Recommendation: build **a + d** into the core (pin the leaky boundary layers, never overclaim), offer **b** as a per-request option, treat **c** as research. This is the number one product decision and it is leyten's call, flagged loudly.

## What we deliberately reuse vs build

| Layer | Decision |
|---|---|
| GPU kernels, attention, paging, quant | Reuse SGLang. Rebuilding this is insane. |
| Per-node block serving | Thin wrapper over SGLang |
| Inter-stage transport | **Build + own** (this is the wedge) |
| Speculative decoding loop | **Build** (nothing off-the-shelf does this) |
| Scheduler / allocation | Build, light |
| Weights distribution | HF + c0mpute mirror |
| Auth, payments, install, dashboard | Reuse c0mpute |
